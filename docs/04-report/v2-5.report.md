## 1. Feature Overview

### 1.1 Feature Description

**V2-5: B2G Dashboard & Cloud Sync** enables educational institutions to use the Phonics 300 app in classroom settings with teacher visibility and data synchronization.

Key capabilities:
- **Teacher Portal**: Secure login/signup at `/teacher` endpoint
- **Class Management**: Generate unique 6-character class codes for student enrollment
- **Student Enrollment**: Anonymous student access via class code + nickname (no personal data collection)
- **Cloud Synchronization**: Automatic upload of lesson results to Supabase without blocking local learning
- **Dashboard Analytics**: Visual reporting with per-unit completion rates and score distribution
- **Graceful Degradation**: Full offline functionality when cloud is unavailable

## 2. Design Overview

### 2.1 Architecture

```
Phonics 300 (Standalone PWA)
├── Frontend: Next.js 16 (App Router)
│   ├── /teacher/page.tsx — Teacher dashboard UI
│   └── /lesson/[unitId]/page.tsx — Student learning (unchanged)
│
├── Backend: Supabase (optional cloud tier)
│   ├── Auth: Supabase Authentication (email/password)
│   ├── Storage:
│   │   ├── teacher_profiles (display_name, school_name, class_code)
│   │   ├── student_profiles (device_id, class_code, nickname)
│   │   └── lesson_logs (student_id, unit_id, score_percent, word_results, etc.)
│   └── Access: Anon Key (NEXT_PUBLIC_*) for client-side auth
│
└── Sync Strategy: Fire-and-forget (non-blocking)
    └── lessonService.ts:syncLessonToCloudIfConnected()
        └── Uploaded at step 5 of saveLessonResults(), independent of local learning
```

### 2.2 Data Model

#### Teacher Profile
```typescript
interface TeacherProfile {
  id: string;           // Supabase auth.users.id
  email: string;
  display_name: string; // E.g., "Ms. Lee"
  school_name?: string; // E.g., "Seoul National Elementary"
  class_code: string;   // 6-char code: A8F3-K9
  created_at: string;   // ISO date
}
```

#### Student Profile (Anonymous)
```typescript
interface StudentProfile {
  id: string;           // UUID
  class_code: string;   // Teacher's code (linking)
  device_id: string;    // Derived from crypto.randomUUID() + localStorage
  nickname: string;     // E.g., "학생 3" (user-provided)
  created_at: string;
}
```

#### Cloud Lesson Log
```typescript
interface CloudLessonLog {
  id?: string;          // UUID (auto-generated by Supabase)
  student_id: string;
  unit_id: string;
  completed_steps: string[]; // E.g., ["sound_focus", "blend_tap", "decode"]
  word_results: Record<string, { attempts: number; correct: number }>;
  duration_minutes: number;
  score_percent: number;
  synced_at: string;    // ISO date
}
```

#### Class Progress Summary (Query Result)
```typescript
interface ClassProgress {
  student_id: string;
  nickname: string;
  completed_units: number;    // Count of finished units
  total_lessons: number;      // Count of completed lessons
  avg_score: number;          // Average score_percent across all lessons
  last_active: string;        // Latest synced_at timestamp
}
```

### 2.3 API Specification

#### Authentication

| Function | Signature | Description |
|----------|-----------|-------------|
| `signUpTeacher()` | `(email, password, displayName?, schoolName?) => Promise<void>` | Create teacher account via Supabase Auth |
| `signInTeacher()` | `(email, password) => Promise<void>` | Login teacher (sets session cookie) |
| `signOutTeacher()` | `() => Promise<void>` | Logout + clear session |
| `getTeacherProfile()` | `() => Promise<TeacherProfile \| null>` | Fetch logged-in teacher data + auto-generate class code if missing |

#### Class Code Management

| Function | Signature | Description |
|----------|-----------|-------------|
| `generateClassCode()` | `(length: 6) => string` | 6-character random code excluding 0/O/1/I/L (31-char alphabet) |
| `getLocalStudentId()` | `() => string \| null` | Retrieve device-based student ID from localStorage |
| `getDeviceId()` | `() => string` | Generate/retrieve unique browser fingerprint via UUID + localStorage |

#### Student Enrollment (Anonymous)

| Function | Signature | Description |
|----------|-----------|-------------|
| `joinClassWithCode()` | `(classCode, nickname) => Promise<StudentProfile>` | Anonymous student joins class via code + nickname |
| (Dedup logic) | Internal | Prevent duplicate enrollments: check `student_profiles` by (class_code, device_id) |

#### Dashboard Queries

| Function | Signature | Description |
|----------|-----------|-------------|
| `getClassStudents()` | `(classCode) => Promise<ClassProgress[]>` | All students in class with aggregated metrics |
| `getUnitCompletionStats()` | `(classCode) => Promise<{unit_id, completion_pct, lesson_count}[]>` | Per-unit completion rate (% of enrolled students who completed unit) |

#### Cloud Sync

| Function | Signature | Description |
|----------|-----------|-------------|
| `syncLessonToCloud()` | `(studentId, lessonLog: CloudLessonLog) => Promise<void>` | Upload lesson result to Supabase |
| `syncLessonToCloudIfConnected()` | `(outcome) => void` | Wrapper: fire-and-forget cloud upload (non-blocking) |
| `isCloudEnabled()` | `() => boolean` | Check if NEXT_PUBLIC_* env vars are set |

### 2.5 Error Handling

| Scenario | Behavior | Impact |
|----------|----------|--------|
| Cloud unavailable (env vars missing) | Show info page with setup guide | Users proceed offline or configure cloud |
| Network timeout during sync | Caught + logged to console.warn | Local learning continues unaffected |
| Supabase auth error | Error message displayed in AuthForm | User retries or contacts support |
| Duplicate student enrollment | Dedup query prevents double-insert | Seamless re-join with same device |

### 2.6 Security Considerations

| Concern | Mitigation |
|---------|-----------|
| Exposed API keys | Only `NEXT_PUBLIC_*` (anon key) used; no server secrets in client |
| Student data privacy | Collect only nickname + device ID; no names, emails, or contact info |
| Class code brute-force | 6 chars × 31-letter alphabet = ~887 million combinations (non-sequential) |
| localStorage persistence | Device ID stored locally; survives browser restarts but not cache clear |
| SSR exposure | `typeof window === 'undefined'` guard in `getSupabase()` |

### 3.4 Database Schema (Supabase)

Three tables required:

#### 1. teacher_profiles
```sql
CREATE TABLE teacher_profiles (
  id UUID PRIMARY KEY REFERENCES auth.users(id),
  email TEXT UNIQUE NOT NULL,
  display_name TEXT NOT NULL,
  school_name TEXT,
  class_code TEXT UNIQUE NOT NULL,
  created_at TIMESTAMP DEFAULT now()
);
```

#### 2. student_profiles
```sql
CREATE TABLE student_profiles (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  class_code TEXT NOT NULL REFERENCES teacher_profiles(class_code),
  device_id TEXT NOT NULL,
  nickname TEXT NOT NULL,
  created_at TIMESTAMP DEFAULT now(),
  UNIQUE(class_code, device_id)
);
```

#### 3. lesson_logs
```sql
CREATE TABLE lesson_logs (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  student_id UUID NOT NULL REFERENCES student_profiles(id),
  unit_id TEXT NOT NULL,
  completed_steps TEXT[] NOT NULL,
  word_results JSONB NOT NULL, -- e.g., {"cat": {"attempts": 2, "correct": 1}}
  duration_minutes INTEGER NOT NULL,
  score_percent FLOAT NOT NULL,
  synced_at TIMESTAMP DEFAULT now()
);
```

Row-level security (RLS) policies:
- Students can only read/write their own records (device_id match)
- Teachers can read all records for their class_code
- Anon key has limited write access (insert only, no delete)

## 7. Lessons Learned

### 7.1 What Went Well

1. **Modular Architecture**: Separation of concerns (supabaseClient.ts handles data, teacher/page.tsx handles UI) made testing and debugging straightforward. No circular dependencies.

2. **Privacy-First Design**: Anonymous enrollment (nickname + device ID) eliminates GDPR/FERPA concerns while still enabling classroom use. Teachers don't need to collect student names.

3. **Fire-and-Forget Sync**: Non-blocking cloud upload means network failures don't interrupt learning. Students continue locally even if Supabase is down. This is the right call for a PWA.

4. **Graceful Degradation**: The cloud-unavailable page (when env vars are missing) provides clear next steps instead of silent failure. Makes deployment much easier.

5. **Data-Driven Charts**: Using aggregated SQL queries (unit completion %, score distribution) instead of client-side calculations keeps dashboard fast even with 100+ students.

### 7.2 Areas for Improvement

1. **N+1 Query Pattern**: `getClassStudents()` loops per-student to fetch lesson logs. For classes >100 students, consider Supabase RPC or materialized view. Current approach is acceptable but not optimal at scale.

2. **Missing `.env.example`**: Should document required Supabase variables earlier. Added to recommendations but not blocking.

3. **RLS Policy Complexity**: Teacher RLS policies (can read own class only) require careful design. Future rounds should document these policies in a comments block.

4. **Import Style Inconsistency**: lessonService.ts mixes relative (`./`) and absolute (`@/`) imports post-Round 11. Minor cosmetic issue but affects linting.

### 7.3 Architectural Insights

- **Starter-Level Adequacy**: Even at "Starter" project level, adding a simple cloud tier works well. No need for advanced patterns (caching, queuing, event sourcing). Supabase handles the heavy lifting.

- **Anon Key Safety**: Using only `NEXT_PUBLIC_*` keys is counterintuitive but works because Supabase RLS policies enforce permissions at the database layer, not the SDK layer.

- **Device ID > User ID**: For a kids' app, device-based identification is better than email/password for students. Simpler, more private, and sufficient for classroom use.
